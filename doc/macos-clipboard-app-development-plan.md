# MacOS 剪切板应用开发技术规划

## 1. 项目目标

本项目要开发一个适用于当前 macOS 系统的剪切板管理应用，核心技术栈为 `Swift + SwiftUI + AppKit + Core Data / SQLite`。

目标不是做一个“能记录剪切板”的简单工具，而是做一个适合长期使用、稳定、低打扰、可扩展的桌面应用。

### 1.1 核心价值

- 自动记录历史剪切板内容
- 快速检索与回贴
- 支持常见内容类型，如文本、富文本、图片、文件路径
- 常驻后台，尽量不打扰用户
- 数据本地存储，优先保护隐私

### 1.2 产品定位

- 面向个人生产力工具
- 默认轻量、低资源占用
- 优先本地离线能力
- 后续可扩展同步、标签、快捷动作、规则过滤等能力

## 2. 技术选型

### 2.1 前端与交互层

- `SwiftUI`：负责主界面、设置页、列表页、搜索页等
- `AppKit`：负责菜单栏常驻、全局快捷键、剪切板监听、窗口控制、系统级交互

### 2.2 数据层

推荐优先采用 `Core Data + SQLite` 组合。

原因：

- 苹果原生支持较好
- 与 Swift 生态结合紧密
- 适合做结构化剪切板记录、索引、过滤与同步扩展

若后期发现数据模型更偏“轻 ORM + 明确 SQL 控制”，可考虑直接使用 SQLite 方案，但第一版建议先以 Core Data 为主，减少底层实现成本。

### 2.3 其他建议依赖

- `UniformTypeIdentifiers`：识别剪切板内容类型
- `Pasteboard` 相关 API：监听与读取剪切板
- `Combine` 或 `async/await`：处理状态流与后台任务
- `Foundation`：持久化、文件处理、编码解码

## 3. 目标系统与兼容策略

### 3.1 系统支持建议

建议以你当前系统版本为目标进行开发与验证。

当前系统版本：

- `macOS 13.2.1`

兼容策略建议：

- 以 `macOS 13.2.1` 为第一优先验证环境
- 如后续需要扩展兼容范围，再逐步测试更高版本 macOS
- 所有系统 API 和界面行为以当前系统实际表现为准

### 3.2 兼容原则

- 优先使用苹果官方原生 API
- 尽量避免依赖系统私有接口
- 与菜单栏、权限弹窗、快捷键等系统行为保持一致
- 在不同分辨率、深色模式、缩放设置下保持可用

## 4. MVP 范围

第一版建议只做“高频可用”的核心能力。

### 4.1 必做功能

- 剪切板自动监听
- 历史记录列表
- 搜索历史记录
- 点击即复制回剪切板
- 菜单栏入口
- 设置窗口
- 本地持久化
- 基础去重策略
- 基础黑名单规则

### 4.2 暂缓功能

- 多设备同步
- 云端备份
- OCR
- 内容摘要
- AI 智能分类
- 团队共享剪切板
- 多窗口复杂工作台

## 5. 应用架构规划

### 5.1 推荐架构

建议采用“分层 + 状态驱动”的结构：

- `Presentation Layer`：SwiftUI 视图
- `App Coordination Layer`：App 启动、窗口管理、菜单栏控制、快捷键分发
- `Domain Layer`：剪切板记录、规则、搜索、标签等业务逻辑
- `Data Layer`：Core Data / SQLite 持久化
- `System Integration Layer`：Pasteboard、权限、文件系统、快捷键

### 5.2 模块划分

建议拆成以下模块：

- `App`
- `ClipboardMonitor`
- `ClipboardStore`
- `Search`
- `Settings`
- `Hotkeys`
- `MenuBar`
- `Permission`
- `Exporter`（后续扩展）

## 6. 功能设计

### 6.1 剪切板监控

实现方式建议：

- 通过 `NSPasteboard.general` 定时轮询或监听变化
- 比较 `changeCount` 判断剪切板是否更新
- 对不同内容类型进行解析并标准化存储

需要处理的内容类型：

- 普通文本
- 富文本
- 图片
- URL
- 文件路径 / 文件列表
- 可能的自定义格式

### 6.2 历史记录管理

每条记录建议包含：

- 唯一 ID
- 内容类型
- 内容摘要
- 原始内容或引用路径
- 创建时间
- 应用来源
- 标签
- 是否置顶
- 是否忽略

### 6.3 搜索

搜索建议分两层：

- 先做本地字段过滤
- 再根据内容类型做全文匹配

后期可扩展：

- 标题/摘要搜索
- 标签搜索
- 应用来源搜索
- 时间范围筛选

### 6.4 回贴与恢复

点击某条历史记录后：

- 写回系统剪切板
- 记录最近一次使用时间
- 支持快捷键快速回贴

### 6.5 过滤与去重

建议加入基础规则：

- 过滤空内容
- 过滤过短内容
- 过滤密码字段
- 过滤特定应用来源
- 相同内容短时间重复只保留一条

## 7. 数据模型建议

### 7.1 核心实体

建议至少包含以下实体：

- `ClipboardItem`
- `ClipboardCategory` 或 `ClipboardTag`
- `AppSource`
- `Setting`

### 7.2 ClipboardItem 字段建议

- `id`
- `contentType`
- `contentText`
- `contentDataPath`
- `previewText`
- `sourceAppName`
- `sourceBundleId`
- `createdAt`
- `updatedAt`
- `lastUsedAt`
- `isPinned`
- `isIgnored`
- `hashValue`

### 7.3 存储策略建议

#### 方式 A：Core Data 管理元数据，文件系统保存大内容

适合：

- 图片
- 大文本
- 富文本
- 二进制内容

优点：

- 查询快
- 元数据和内容分离
- 对大内容更稳妥

#### 方式 B：全部进 SQLite / Core Data BLOB

适合：

- 早期快速开发
- 内容规模不大

风险：

- 大图片和大文本容易让数据库膨胀

建议：

- 第一版采用“元数据入库，大内容落盘”的混合策略
- 数据库只存索引与引用路径

## 12. 风险点与规避方案

### 12.1 剪切板轮询开销

风险：

- 频繁轮询可能消耗资源

规避：

- 降低轮询频率
- 合并重复事件
- 做内容哈希去重

### 12.2 图片和大文本占用空间

风险：

- 数据库膨胀
- 启动变慢

规避：

- 大内容文件落盘
- 数据库只保存索引与预览

### 12.3 系统权限和体验问题

风险：

- 权限提示影响首次体验
- 快捷键与系统冲突

规避：

- 首次启动时给出清晰引导
- 快捷键可配置

### 12.4 隐私安全

风险：

- 剪切板可能包含敏感信息

规避：

- 默认本地存储
- 提供敏感词过滤
- 提供忽略应用和忽略规则

## 13. 第一版交付标准

满足以下条件即可视为 MVP 完成：

- 应用可以常驻后台
- 剪切板自动记录
- 可以搜索历史记录
- 可以一键回贴
- 数据可以持久化
- 有基础设置项
- 菜单栏可访问
